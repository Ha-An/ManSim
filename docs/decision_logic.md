# 의사결정 로직

## 목적
현재 시스템의 목적은 남은 기간 안에 완제품 마감 수를 최대화하는 것입니다.

## 모드별 개요
### fixed_priority
- 고정된 작업 우선순위를 사용합니다.
- 가장 빠르고 재현성이 높습니다.
- 상황별 유연성은 낮습니다.

### adaptive_priority
- 상태 요약을 바탕으로 규칙 기반 우선순위를 조정합니다.
- LLM 없이도 일부 상황 반응이 가능합니다.
- 복잡한 병목 전환을 해석하는 능력은 제한적입니다.

### llm_planner
- `MANAGER_BOTTLENECK_DETECTOR`가 현재 병목을 진단합니다.
- `MANAGER_DAILY_PLANNER`가 병목 진단과 실행 상태를 함께 보고 하루 계획을 수립합니다.
- 작업자 큐가 있으면 작업자는 이를 우선 소비하고, 불가능할 때만 로컬 `priority_score`로 fallback합니다.

## 현재 LLM 매니저 구조
### 병목 탐지기
- 입력: 현재 관측, 흐름 마감 신호, 제약 신호, 보조 상세
- 출력: 상위 병목, 후보 작업군, 근거 추적
- 역할: 완제품 마감을 가장 제한하는 제약을 2~3개로 압축 진단

### 일일 계획기
- 입력: 실행 상태, 마감 신호, 제약 신호, 작업자별 현재 큐, 병목 탐지기 가설
- 출력: 당일 계획 diff, 작업자별 큐, 가중치 조정, detector 정렬 여부
- 역할: 오늘 기준으로 가장 효과적인 작업 개입을 선택하고 작업자 행동으로 연결

## 평가 포인트
- 병목 진단이 실제 종료 상태와 일치하는가
- 계획이 실제 작업자 행동을 바꾸는가
- `manager_queue` 사용 비율이 의미 있게 존재하는가
- detector 오판 시 planner가 `follow`가 아니라 `partial_override` 또는 `override`를 내는가
