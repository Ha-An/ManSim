# LLM 프롬프트 설계 원칙

## 공통 원칙
- 프롬프트를 계속 덧붙여 길게 만들지 않습니다.
- 약한 문장을 지우고, 더 짧고 강한 문장으로 교체합니다.
- 자연어 장문보다 구조화된 입력 순서와 의미를 먼저 정리합니다.

## 병목 탐지기 설계 원칙
- 현재 상태에서 가장 중요한 병목을 다시 랭킹하는 문제로 정의합니다.
- 눈에 띄는 기계 이상 나열이 아니라 완제품 마감 제약 랭킹에 집중시킵니다.
- 입력은 `throughput_closure_state`, `constraint_state`, `supporting_detail` 순으로 정리합니다.
- 출력은 상위 병목, 후보 작업군, 근거 추적 중심으로 짧게 유지합니다.

## 일일 계획기 설계 원칙
- detector를 절대 진실이 아니라 오늘 검증할 가설로 취급합니다.
- 오늘 실행 증거가 더 강하면 detector를 부분 또는 전체 override할 수 있어야 합니다.
- 일반 가중치 변경보다 구체적인 작업자 큐를 우선합니다.
- `maintain`은 강한 근거가 있을 때만 허용합니다.

## 피해야 할 것
- 규칙을 계속 프롬프트에 덧붙여 instruction debt를 쌓는 것
- machine-local anomaly를 planner와 detector 양쪽에 동시에 과도하게 강조하는 것
- detector가 작업자별 queue까지 짜게 만드는 것
